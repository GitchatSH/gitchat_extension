- 2026-04-22: Fix GitchatSH/gitchat_extension#175 topic pagination `hasMore=false` (`gitstar-webapp/backend`, branch `vincent-fix-topic-pagination-hasmore` off `develop`). Slug reported `GET /messages/conversations/:parentId/topics/:topicId/messages?limit=30` returned `{ messages:[30], nextCursor:null }` despite topics holding >30 messages — FE pagination (`use-message-thread.ts:80` derives `hasMore = !!nextCursor`) broke scroll-up. Both the topics alias endpoint (`topics.controller.ts:197-217`) and the regular `/conversations/:id` endpoint (`messages.controller.ts:211-235`) call the exact same `MessagesService.getMessages(...)`, so logic divergence was ruled out early. **Root cause**: the single query combined `.leftJoinAndMapMany` (reactions + attachments, both 1:N) + `.leftJoinAndMapOne` (self-joined `messages` via `replyToId`) + `.take(pageLimit+1)`. TypeORM 0.3.28's subquery strategy for `.take()` should normalise the cartesian product, but in practice for topic data profiles (replies + reactions chained) LIMIT was landing on joined rows and clipping distinct root entities below `pageLimit+1` — so `hasMore = messages.length > pageLimit` evaluated false even when the table had more. DM/group didn't repro because their data profile rarely triggers the clip. **Fix**: split into two-phase query. Phase 1: `.select(['m.id','m.createdAt'])` — no joins, applyMessagesScope + deletedAt filter + cursor filter + `.take(pageLimit+1)` → LIMIT now reliably applies to root rows. Phase 2: build a second qb with the same three leftJoinAndMap* calls but use `WHERE m.id IN (:...ids)` (no LIMIT, so cartesian explosion is harmless), preserving the same `orderBy` direction. `hasMore` is now computed off phase-1 id rows, not the hydrated result. Scope preserved: aroundDate branch, legacyParentId UNION scope, soft-delete filter, read cursors, direction-based reverse — all untouched. **Test strategy**: wrote a "SQL-aware" mock in `messages.service.spec.ts` that inspects whether the qb was built with joins AND take (old pattern → returns `take-1` rows to model the clip) vs no-joins+take (new id query → returns accurate `take` rows) vs where-in-ids (hydration → returns matching rows). 2 new tests (31 messages → `nextCursor` non-null; 30 messages → `nextCursor` null) FAILED on pre-fix code with `expected not null, received null`, PASS after refactor.
